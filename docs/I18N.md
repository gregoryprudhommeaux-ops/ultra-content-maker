# I18N — ULTRA CONTENT MAKER

## Locales

| Code | Role |
|------|------|
| `en` | **Default** — source of truth for all UI strings |
| `fr` | Auto-generated from EN |
| `es` | Auto-generated from EN |

## Two language systems

| System | Controls | Mechanism |
|--------|----------|-----------|
| **UI locale** | Buttons, labels, errors, wizard | `next-intl` + `messages/*.json` |
| **Content language** | LinkedIn posts, brain text | Client field `content_language` + LLM prompts |

Do not conflate them. A user with UI in French can generate English posts for a US client.

## File structure

```
messages/
  en.json    # edit manually — source of truth
  fr.json    # generated via npm run i18n:sync
  es.json    # generated via npm run i18n:sync
```

## Key naming convention

Dot-separated, feature-first:

```
auth.login.title
auth.login.submit
dashboard.clients.empty.title
dashboard.clients.empty.cta
onboarding.step1.title
brain.validate.button
generate.calendar.disabledHint
errors.llm.timeout
```

## Workflow for developers

1. Add new keys **only** to `messages/en.json`
2. Use `const t = useTranslations('dashboard')` in components — no hardcoded UI strings
3. Run `npm run i18n:sync` to refresh `fr.json` and `es.json`
4. Spot-check FR/ES in browser via language switcher
5. Before release: full journey in `en`, `fr`, `es` — no missing keys (next-intl shows key name if missing)

## Routing

- URLs: `/[locale]/clients`, etc.
- Middleware: locale detection + auth
- Fallback: `en`
- Preference: `users/{uid}.preferredLocale` + cookie

## i18n:sync script behavior

- Input: `messages/en.json`
- Output: merge into `fr.json`, `es.json`
- Use OpenAI: preserve keys, placeholders `{name}`, and JSON structure
- Do not translate: brand name "ULTRA CONTENT MAKER", "Content Brain" (product terms OK to keep or glossarize consistently)

## QA checklist

- [ ] Language switcher updates URL and persists preference
- [ ] Login, client list, wizard step 1 — EN/FR/ES
- [ ] No raw i18n keys visible in UI
- [ ] Generated post language matches `content_language`, not UI locale

## Out of scope v1

- Runtime per-string API translation
- Locales beyond en/fr/es
- Translating generated LinkedIn content via i18n files (LLM handles that)
